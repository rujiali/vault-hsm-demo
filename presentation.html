How to protect sensitive data without ever exposing the key

Situation

Your organisation handles sensitive data that must be protected

  • Personnel records, financial data, credentials, health information
  • ISM, Privacy Act, and PCI-DSS mandate encryption of sensitive data at rest
  • Encryption is already in place — or required to be
  • The perimeter is no longer the boundary — data protection must travel with the data

The given

Encryption is non-negotiable. Regulators, auditors, and boards all agree. The question is not whether to encrypt — it is how safely the keys are managed.

Establish shared context the audience already agrees with. Encryption is mandated and happening. The problem is not the encryption itself — it's what happens to the keys.

Complication

Encryption is only as strong as key management

  • Keys end up in config files, environment variables, and logs
  • No audit trail — you cannot prove who accessed what, or when
  • Key rotation requires application changes and risks downtime
  • A single privileged user can export or delete a key unilaterally
  • Compliance requires hardware-backed key storage

The real risk

Whoever holds the key holds the data. If an attacker — or an insider — gets the key, the encryption is worthless.


And right now, the key is probably somewhere it shouldn't be.

This is where the audience leans forward. The encryption they already have is undermined by poor key management. Name the specific failure modes — config files, no audit, single person with full power.

How do you encrypt data at scale — without the application ever touching a key?

Key Protection

Where Does the Encryption Key Live?

Vault-Managed Keys

Transit and FPE keys live in Vault's storage — but stored as ciphertext. Without the HSM, they are unreadable.

HSM Master Key

Lives inside the HSM hardware — never exported. Used to wrap and unwrap all Vault-managed keys on demand.

The Result

Even if someone steals Vault's storage, they get encrypted keys they cannot unwrap — the HSM is the only way to use them.

Vault Transit & Transform

Choosing the Right Encryption Method

Method Engine Output Schema change Searchable Best for
Encryption (AES-GCM) Transit vault:v1:AbC… — longer than input Yes ✗ random IV JSON, blobs, documents
FPE (FF3-1) Transform Same format as input — 4111-2345-6789-0123 No ✓ deterministic Tax IDs, national IDs — legacy systems that require original format
Tokenization Transform Random surrogate token — tok_8f3kLp2q Yes ✓ deterministic Card numbers, PCI compliance — token has no relation to original
Masking Transform Redacted display — ****-****-****-1234 No ✗ irreversible Logs, display layer, non-prod data
Why Transit cannot be searched

AES-GCM generates a fresh random IV for every encryption — the same value produces a different ciphertext every time. A DB index is useless.

Why FPE & tokenization can be searched

Both are deterministic — the same input always produces the same output for the same key. DB equality indexes work unchanged.

Transit is for encrypting arbitrary data where you don't need to search. FPE is for structured sensitive fields where schema changes are undesirable. Tokenization is for card number or ID surrogate replacement. Masking is one-way — display only.

Encrypting an Existing Database

Encrypting an existing table — FPE in-place

1
Mount Transform engine, define FPE transformations

One transformation per sensitive column — each with its own alphabet and format pattern. No application code changes yet.

2
Batch-encode all rows via Vault

transform/encode/role accepts thousands of values per call. Entire table encrypted in one pass — no downtime.

3
Write encrypted values back to the same columns

Format is preserved — no schema change, no column rename, no index rebuild. Existing queries continue to work.

Search — encrypt the query term first

FPE or tokenization — equality search
User searches: "123456789"
↓ app encrypts the search term via Vault
Vault FPE: "123456789" → "847239156" (always same)
↓ query the encrypted column directly
SELECT * FROM employees
WHERE tfn_enc = '847239156'
↓ decrypt result to display
✓ Row returned — app decrypts for display. DB never saw plaintext.
⚠ Range queries (BETWEEN, >, <) are not supported on encrypted columns — accepted trade-off for sensitive fields.

FPE encrypts in-place — same column, same format, no schema change. The batch endpoint handles the whole table in one pass. For search: you never query for plaintext — encrypt the search term first via Vault, then run the WHERE clause on the encrypted column.

Key Rotation

Retiring an Old Key Version

Why Retire?

Rotation keeps old versions available by default. Retirement is the deliberate act of enforcing that no data encrypted with a specific version can be decrypted — meeting compliance mandates for key lifetimes.

Two-step process

1
Rewrap all ciphertext
Vault re-encrypts each record with the current key version — application never sees plaintext.
2
Advance min_decryption_version
Vault rejects any ciphertext older than this version — old key material is permanently inaccessible.

Impact on Ciphertext

The version prefix changes: vault:v1:… → vault:v3:…. The ciphertext value changes (fresh IV). The plaintext is identical — zero data loss.

CLI — Rewrap & Retire
# 1. Rewrap a record to the current key version
vault write transit/rewrap/demo-key \
  ciphertext="vault:v1:ABC…"

# Returns a new ciphertext at the latest version
ciphertext: vault:v3:XYZ…

# 2. Retire v1 — v1 ciphertext now permanently rejected
vault write transit/keys/demo-key/config \
  min_decryption_version=2

# Proof — attempt to decrypt old ciphertext
vault write transit/decrypt/demo-key \
  ciphertext="vault:v1:ABC…"
Error: ciphertext version is not allowed

Key point for the audience: rewrap is a server-side operation — the application submits ciphertext, Vault returns new ciphertext, plaintext never travels. After advancing min_decryption_version, that old version is mathematically unrecoverable even if someone exfiltrated the ciphertext earlier.

Key Governance

Who Controls the Keys?

1 — Key Configuration

Set at creation — exportable=false, deletion_allowed=false. Immutable. Even root cannot override.

2 — ACL Policies

Controls who can call each API path. Key viewer, exporter, destroyer — all separate roles with least privilege.

3 — Sentinel Policy (EGP)

Controls under what conditions. Time of day, IP range, MFA. Applied at the API endpoint — even privileged tokens are subject to it. (Available in Vault Enterprise — not in live demo)

4 — Control Groups ★ Demo

Requires a second human to approve before Vault releases key material. No single person can export a key unilaterally. Three-phase flow: operator requests → custodian approves → operator collects.

Control Groups

Dual Control — No Unilateral Action

1
Requestor calls export endpoint

Vault does not return the key. Returns a wrapping token — request held pending approval.

2
Requestor shares the wrap accessor with custodian

Out of band — email, Slack, ticket. Custodian receives the accessor (opaque reference), not the wrap token itself.

3
Custodian authorises via Vault

Calls sys/control-group/authorize with the accessor — returns {} HTTP 200. Custodian identity recorded in audit log.

4
Requestor collects key material

Unwraps the token (HTTP 204, approval consumed), then calls export to retrieve actual key material.

✓ Every export is a two-person event — enforced by Vault, recorded in audit log.

The wrapping token is the mechanism — the requestor gets a token, not the secret. It's only redeemable after approval. Both identities appear in the audit log — this is your 4-eyes compliance evidence.

Access Control

Separation of Duties

Different teams get different capabilities — enforced by Vault policy, not application logic.

Encrypt Only

Application can submit plaintext and receive ciphertext.


Cannot decrypt. Cannot manage keys.

Decrypt Only

Authorised service can decrypt ciphertext to retrieve data.


Cannot encrypt. Cannot manage keys.

Key Admin

Security team can rotate and manage keys.


Cannot encrypt or decrypt data.

Separation of duties is enforced at the policy layer — not the application layer. The app team literally cannot get a token with decrypt rights. Strong compliance posture.
