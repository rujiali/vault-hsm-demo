The Challenge

How do you protect a key
without seeing it?

  • Applications need to encrypt sensitive data
  • Keys must never be exposed to applications
  • Compliance requires hardware-backed key storage
  • Key rotation must not break existing data
  • Separation of duties — who can encrypt vs decrypt

The Risk

When applications manage their own keys, keys end up in config files, environment variables, and logs — exposed and unaudited.


The application should never see the key.

The core problem: key management is hard. When apps handle their own keys, they always end up somewhere they shouldn't. Goal: move key management out of the application entirely.

Key Protection

Where Does the Encryption Key Live?

Transit Key (AES-256)

Lives in Vault's storage — but stored as ciphertext. Without the HSM, it is unreadable.

HSM Master Key

Lives inside the HSM hardware — never exported. Used to wrap and unwrap the Transit key on demand.

The Result

Even if someone steals Vault's storage, they get encrypted keys they cannot unwrap — the HSM is the only way to use them.

Runtime Flow

Startup vs. Per Request

STARTUP — once only
1
vault-main starts — sealed

All storage encrypted at rest. Cannot serve requests.

2
→ vault-hsm: unwrap barrier key

3
Transit keyring loaded into memory

Barrier key decrypts storage. demo-key cached in memory. vault-main is unsealed.

🔑 Master key never leaves vault-hsm. Transit keys never touch disk in plaintext.
PER REQUEST — every call
1
App → vault-main: plaintext + token

App calls POST /v1/transit/encrypt/demo-key. Token validated against policy.

2
vault-main encrypts using in-memory key

demo-key already in memory. vault-hsm not called. Fast and local.

3
App receives ciphertext

vault:v1:... returned. Key never written to disk. App never saw the key.

⚡ vault-hsm re-engaged only on restart or key rotation — not on every request.

Key performance point: vault-hsm is only called once at startup. Every subsequent encrypt/decrypt is served from memory — no HSM round-trip on the hot path. Addresses the "won't this be slow?" question.

HSM Integration

Hardware Root of Trust

🔐

Transit Seal

Vault's Transit seal allows one Vault instance to act as the root of trust for another — the same pattern as a hardware HSM, implemented in software.

🏛️

Auto-Unseal

Vault unseals automatically on restart by calling the HSM. No human operator needed. Master key never exists on Vault's disk.

🛡️

Seal Wrap

Every sensitive value in Vault storage is wrapped by the master key. Vault's storage is useless without the HSM.

Key Lifecycle

Key Rotation Without Breaking Data

  • Keys are versioned — rotation creates a new version
  • New encryptions use the latest key version
  • Old ciphertext remains decryptable — no re-encryption required
  • Minimum decryption version can be set to retire old keys
  • Zero application downtime during rotation
# Rotate the key
vault write -f transit/keys/demo-key/rotate

# New ciphertext uses v2
vault:v2:XYZ789abc...

# Old v1 ciphertext still works
vault:v1:ABC123xyz... ✓

Key rotation is additive — old versions are retained. The version prefix (v1, v2...) is embedded in ciphertext so Vault always knows which key to use. Apps don't change. No downtime.

Key Rotation

How Apps Keep Working After Rotation

Rotation is Additive

A new key version is created — old versions are kept, not deleted. Vault maintains all versions until explicitly retired.

Ciphertext is Self-Describing

The key version is embedded in every ciphertext token. Vault reads it and uses the correct version automatically.

vault:v1:ABC123...  ← uses v1
vault:v2:XYZ789...  ← uses v2

Zero App Changes

Apps send ciphertext to Vault — Vault selects the right key version transparently. No code change. No downtime.

Key Governance

Who Controls the Keys?

1 — Key Configuration

Set at creation — exportable=false, deletion_allowed=false. Immutable. Even root cannot override.

2 — ACL Policies

Controls who can call each API path. Key viewer, exporter, destroyer — all separate roles with least privilege.

3 — Sentinel Policy (EGP)

Controls under what conditions. Time of day, IP range, MFA. Applied at the API endpoint — even privileged tokens are subject to it.

4 — Control Groups

Requires a second human to approve. No single person can export or delete a key unilaterally. Dual control enforced by Vault.

Control Groups

Dual Control — No Unilateral Action

1
Requestor calls export endpoint

Vault does not return the key. Returns a wrapping token — request held pending approval.

2
Requestor sends wrapping token to custodian

Out of band — email, Slack, ticket. Custodian reviews the request.

3
Custodian authorises via Vault

POST /v1/sys/control-group/authorize — custodian identity recorded in audit log.

4
Requestor unwraps — gets the key

vault unwrap <wrapping_token> — only succeeds after approval. Full audit trail.

✓ Every export is a two-person event — enforced by Vault, recorded in audit log.
# Policy with control group
path "transit/export/encryption-key/demo-key" {
  capabilities = ["read"]
  control_group = {
    ttl = "1h"
    factor "authorise" {
      identity {
        group_names = ["key-custodians"]
        approvals = 1
      }
    }
  }
}
# Sentinel EGP — business hours only
import "time"

main = rule {
  time.now.hour >= 9 and
  time.now.hour < 17
}

# Applied to: transit/export/*
# Enforcement: hard-mandatory

The wrapping token is the mechanism — the requestor gets a token, not the secret. It's only redeemable after approval. Both identities appear in the audit log — this is your 4-eyes compliance evidence.

Access Control

Separation of Duties

Different teams get different capabilities — enforced by Vault policy, not application logic.

Encrypt Only

Application can submit plaintext and receive ciphertext.


Cannot decrypt. Cannot manage keys.

Decrypt Only

Authorised service can decrypt ciphertext to retrieve data.


Cannot encrypt. Cannot manage keys.

Key Admin

Security team can rotate and manage keys.


Cannot encrypt or decrypt data.

Separation of duties is enforced at the policy layer — not the application layer. The app team literally cannot get a token with decrypt rights. Strong compliance posture.
